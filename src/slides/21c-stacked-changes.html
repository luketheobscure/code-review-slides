Keep changes small.

  • 200–400 line PRs have 40% fewer defects and are approved 3× faster 1,2
  • Defect detection rates: 87% in small PRs, 28% in massive ones 1
  • You don't have to write small, you can split after.
  • Jujutsu (jj): VCS-level — rethinks commits so splitting and reordering is native, not surgery
  • GitHub native stacked PRs (Apr 2026): platform-level — stack map lives inside the PR UI, no extra tools required 2

1 Smart Bear / Cisco Systems, Code Review at Cisco Systems, 2006
2 GitHub stacked PRs launch, Apr 2026 — InfoQ

An analysis of 1.5 million pull requests found that PRs between 200 and 400 lines had 40% fewer defects and were approved three times faster than larger ones.
Defect detection rates fall from 87% in small PRs to 28% in massive ones
Something to remember when dealing with LLMs, for them "thinking" has the same cost as "writing", so it's trivial to generate a 2,000-line change. Wether or not those 2k lines are even necessary.
The fix isn't writing small — it's splitting after the fact, or even during your workflow. Stack the output into a chain of reviewable slices. Different tools solve this at different levels of the stack. Jujutsu (jj) is a VCS replacement — Mercurial-inspired, built at Google. It rethinks how commits work so that splitting a large change or reordering history feels native rather than git branch surgery. GitHub stacked PRs: stack map in the PR UI, CI targets main, no extra tools needed. Others if asked: Sapling (Meta, niche), Graphite (paid SaaS on top of GitHub). The shape of good review hasn't changed. AI just made ignoring it unaffordable.
