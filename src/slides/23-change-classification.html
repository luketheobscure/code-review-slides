Triage every change.

LOW RISK

Docs · copy · UI

Verified & auto-merged.

MEDIUM RISK

Business logic · public APIs

Review happens post-merge, so developer is not blocked

HIGH RISK

Auth · payments · migrations

Human review required before merge.

Spend human attention where it changes the outcome.

A certain large company has invested heavily in triaging changes by risk level, and routing them to different review queues and processes. The principle is to spend human attention where it changes the outcome. Low-risk changes like docs, copy, and UI can be verified with tests and auto-merged. Medium-risk changes like business logic and public APIs get merged immediately but reviewed asynchronously, so developers aren't blocked. High-risk changes like auth, payments, and migrations require human review before merge. This kind of triage is more important than ever when the volume of changes is increasing dramatically.
